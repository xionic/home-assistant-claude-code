### Notes on the working indicator
Three bouncing dots + "Working…". It appears the moment you send, disappears as
soon as streaming text or a tool card appears, and reappears between tool calls
while Claude decides its next step. It's also restored if you reconnect mid-run.

### Notes on model display
Removing "Default" means the dropdown always shows a concrete model. To keep it
honest, the server now forwards the SDK's reported model (`init.model`) and the
UI selects it — so it reflects what's *actually* running. If you pick a model, that
choice persists and is sent as an override; if you don't, the SDK default is used
and merely displayed.

## Part 2 — Investigations

### 2a. `/usage`

`usage` **is** in the SDK's slash-command list, and sending `/usage` does execute —
but in the SDK's non-interactive (print) mode it only returns a stub line
("You are currently using your subscription to power your Claude Code usage").
The real `/usage` breakdown (subscription limits, weekly/5-hour windows) is an
**interactive TUI component** that the Agent SDK does not render. So it can never
fully work through this web UI as-is.

**What I did:** implemented a local `/usage` that reports *this session's* totals
(messages sent, turns, accumulated cost) from data we already track on each
result. It's clearly labelled as session usage, with a note that the full Claude
Code `/usage` isn't available here.

**If you want the real subscription-usage numbers**, that requires calling
Anthropic's usage API directly (separate from the agent loop) and rendering it
ourselves — feasible but a separate feature.

### 2b. ha-context "history" bug — found

Reproduced by asking the live agent to fetch 24h of history:
- It first tried `date -u -d '24 hours ago'` → **`date: invalid date`**. Alpine's
  BusyBox `date` has no `-d`/`-v` flags, so REST `history/period` URL-building
  (which needs a start timestamp) fails. Claude burned several turns working
  around it with python.
- The REST `history/period` call eventually returned `[]`.

Root cause: the guidance steered Claude to REST + shell `date` math, which is
broken on this Alpine image. Meanwhile **`ha-ws-client` has working history
tooling** that needs no date math:
- `ha-ws-client history <entity>` → state-change history (worked; returned 0
  entries for `sensor.car_12v_battery_voltage` because that sensor is **genuinely
  stale** — stuck at -7.1 V since ~June 7, so no recorded changes).
- `ha-ws-client stats <entity>` → long-term statistics; **returned real data**
  (hourly min/max/mean).
- `ha-ws-client logbook` → recent logbook entries.

**Fix:** documented `history`/`stats`/`logbook` in both the HA context
(`ha-context.sh`) and the skill (`SKILL.md`), with an explicit warning not to
build REST history URLs with `date` on this image. Also noted that an empty
history just means no recorded changes in the window (not an error).

> Side note: the car-battery sensor reading (-7.1 V, last changed ~June 7) looks
> like a dead/disconnected sensor feed — worth checking the Škoda integration.

### 2c. ha-mcp update / Alpine → Debian rebase

**First, the framing question:** ha-mcp is currently **disabled** and fully
replaced by `ha-ws-client` (states/services/templates/history/stats) and
`ha-lovelace` (dashboards). Everything in your test prompts now works without it.
So before investing in a rebase, it's worth deciding whether ha-mcp is still
wanted at all.

**Is Debian actually required?** Likely, but not certainly:
- ha-mcp installs via `uvx` (Python). Newer ha-mcp (7.x) tends to pull deps with
  native extensions (pydantic-core/Rust, cryptography, etc.). On Alpine (musl),
  prebuilt manylinux wheels don't apply, so `uvx` may try to **compile from
  source** — needing a full build toolchain (gcc, rust, headers) and often
  failing or bloating the image. On Debian (glibc) those wheels install cleanly.
- The current Dockerfile also force-installs the **musl** build of the Agent SDK
  binary specifically because the base is Alpine. On Debian we'd switch to the
  default glibc build (simpler).

**Cheapest first step (recommended before any rebase):** test whether
`uvx ha-mcp@7.7.0 --help` even installs/runs on the *current* Alpine image. If it
does, no rebase is needed. I can add a one-line debug probe and check it.

**If a rebase is needed, the plan (≈ half-day, isolated branch):**
1. `build.yaml`: switch `build_from` to HA's Debian base images
   (`ghcr.io/home-assistant/{arch}-base-debian`) — keeps bashio + s6.
2. `Dockerfile`: `apk add` → `apt-get install`; install Node via the Debian base
   or nodesource; install `uv`.
3. Drop the musl-specific Agent SDK pin — let npm install the glibc binary.
4. `ha-ws-client` is a static Go binary → works unchanged on glibc.
5. Re-test: auth/login flow, ha-ws-client, ha-lovelace, /diag, then ha-mcp 7.x.
   Image will be larger (Debian) — acceptable on a Pi 4 with storage.

**Recommendation:** keep the lean ha-ws-client/ha-lovelace approach. Only rebase
if there's a specific ha-mcp 7.x capability you want that we can't replicate with
ha-ws-client/REST. If you name those capabilities, I can likely add them directly.

## Part 3 — Design questions (answers, no action taken)

### Q1. Multi-session support with browsing previous sessions — **Moderate (~1–2 days)**

Today there's one persistent conversation (`/data/conversation.json`, holding a
transcript + SDK resume id). Multi-session is mostly an additive refactor:
- **Backend:** turn the single `conversation` into a map keyed by id; persist each
  to `/data/sessions/<id>.json` plus a lightweight index (id, title, updatedAt);
  add an "active session" pointer; CRUD over WebSocket (list/new/switch/rename/
  delete). Resume already works per `sessionId`, so continuing any session is free.
- **Frontend:** a session-list drawer; click to switch (we already replay a
  transcript via `renderHistory`); rename/delete; titles from the first user
  message.
- **Note:** Claude Code also keeps its own per-session files on disk under the
  config dir, so "browse previous sessions" could even surface sessions created
  elsewhere — but managing our own store is simpler and gives us titles/ordering.

Low risk; the transcript replay and resume machinery already exist. Most of the
work is UI.

### Q2. The "remote control" feature Claude Code has — **Depends on what you mean**

This one needs clarification, because there are a few things it could mean:
- **Access/drive the agent from another device:** already done. This *is* a remote
  web UI — reachable wherever you can reach HA (LAN, VPN, or Nabu Casa remote).
  Multiple clients already stay in sync (the server broadcasts to all connections),
  so you can drive it from your phone and watch on your laptop.
- **Push notifications when the agent finishes or needs a permission:** very
  doable (~hours) — we already can call HA services, so a `notify.*` ping on
  completion / permission-request is easy.
- **Claude Code's own "remote"/cloud teleoperation** (controlling a session via
  Anthropic's backend, cloud agents, phone hand-off): **not replicable** in a
  self-hosted add-on — it relies on Anthropic-hosted infrastructure.

Tell me which of these you mean and I'll scope it precisely.

### Q3. `/resume` — **Easy once Q1 exists (otherwise Moderate)**

`/resume` is essentially the *UI half* of multi-session: pick a past session and
continue it. If we build Q1, `/resume` is a thin command that opens the session
browser — trivial. Standalone (without a session store) it's moderate, because
you still need a list of resumable sessions to show. Best built together with Q1.

### Q4. `/btw` — **Easy to Moderate, depends on exact behaviour**

`/btw` (inject an aside / "by the way" steering message) isn't something I can
confirm as a stable built-in, so treat this as based on the likely intent:
- **"Queue a follow-up to send when the current turn finishes":** easy, pure
  client-side — hold the message and auto-send on the next `result`.
- **True mid-turn injection** (feed a message into a *running* query without
  interrupting): moderate. Our server calls `query()` once per prompt; real
  injection needs the SDK's **streaming-input mode** (an async-iterable prompt you
  can push more user messages into mid-run). That's an architecture change from
  per-prompt `query()` to a persistent streaming session, but it's a documented
  SDK capability and would also unlock nicer interactivity generally.

If you can point me at what `/btw` does in your Claude Code, I'll match it.
